# OmniKube 多集群多租户 K8S 管控平台 - 功能设计书 v2

> v2 修订说明：本版在 v1 基础上修复了三处越权风险（namespace 参数混淆、跨命名空间数据泄露、WebSocket 鉴权缺失），重写了集群连接池与 Casbin 权限模型，并补齐认证、审计、加密、级联清理等工程缺口。变更点在文中以 `【v2】` 标注。

## 1. 项目概述

### 1.1 项目简介

OmniKube 是一个基于 Golang 和 React 构建的企业级**多集群、多租户 Kubernetes 可视化管理平台**。平台核心解决“用户全局统一管理”、“多 K8S 集群动态托管绑定”以及“精细到 Namespace 级别的资源可见性与操作权限隔离”三大痛点。用户只需在 OmniKube 注册一次，即可由系统管理员（Admin）按需分配不同集群、不同 Namespace 的管理或只读权限。

### 1.2 技术栈选型

* **后端 (Backend):** Golang + Gin + Client-go（typed/dynamic/discovery 多客户端）+ Casbin（基于 Domain 的 RBAC）+ GORM
* **认证 (Auth):** 【v2】JWT（Access Token + 短期有效）+ 登录态中间件，统一注入 `user_id`
* **前端 (Frontend):** React + Ant Design + Axios + xterm.js（WebSSH 终端）
* **数据存储:** MySQL / PostgreSQL
* **密钥管理:** 【v2】Kubeconfig 字段使用 AES-256-GCM 加密，主密钥来自环境变量 / 外部 KMS，不落库

## 2. 核心架构与多集群动态连接机制

### 2.1 逻辑架构

OmniKube 独立于 K8S 集群运行，作为统一网关，后端内存维护一个**线程安全的集群连接池**。

```
                   ┌─────────────────────────────────────────┐
                   │            OmniKube 前端 (React)         │
                   └────────────────────┬────────────────────┘
                                        │ (HTTP / WebSocket)
                                        ▼
                   ┌─────────────────────────────────────────┐
                   │            OmniKube 后端 (Go)            │
                   │  ┌──────────┐ ┌──────────────┐ ┌───────┐ │
                   │  │ JWT 认证 │→│ RBAC 鉴权中间件│→│Casbin │ │
                   │  └──────────┘ └──────┬───────┘ └───┬───┘ │
                   │                      │             │     │
                   │                      ▼             │     │
                   │            ┌──────────────────┐    │     │
                   │            │ 集群连接池        │◄───┘     │
                   │            │ (RWMutex + 多Client)│       │
                   │            └─────────┬────────┘          │
                   │            ┌─────────┴────────┐          │
                   │            │ 健康探活 / 审计日志 │         │
                   └────────────┼──────────────────────────────┘
                                │ (动态路由切换)
         ┌──────────────────────┼──────────────────────┐
         ▼                      ▼                      ▼
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│  K8S 集群 F     │    │  K8S 集群 G     │    │  K8S 集群 H     │
└─────────────────┘    └─────────────────┘    └─────────────────┘
```

### 2.2 动态连接池实现原理【v2 重写】

**连接池结构**：每个集群不是单个 `Clientset`，而是一组客户端，以支持 CRD / 任意资源。

```go
type ClusterClient struct {
    Typed      *kubernetes.Clientset   // 内置资源 (Pod/Deployment/Service...)
    Dynamic    dynamic.Interface       // CRD / 任意 GVR
    Discovery  discovery.DiscoveryInterface
    RESTMapper meta.RESTMapper         // GVK <-> GVR 映射
    Config     *rest.Config            // 供 remotecommand(WebSSH) 复用
}

type ClusterPool struct {
    mu      sync.RWMutex
    clients map[string]*ClusterClient  // key = ClusterID
}
```

**生命周期**：

1. Admin 在 Web 界面添加新集群，输入 `ClusterID` / `Name` 并粘贴 `Kubeconfig`。
2. 后端用 `clientcmd` 解析文本，构建上述全套客户端；**入池前先用 `Discovery` 做连通性 + 权限自检**（即 5.1 的“测试连接”）。
3. 通过 `RWMutex` 加写锁写入 `clients`，读请求加读锁，杜绝并发 panic。
4. 【v2】**进程启动时从 DB 全量重建连接池**（解密 Kubeconfig → 建客户端），保证重启 / 多副本扩容后池子可用。
5. 【v2】**后台 goroutine 定时探活**（每 30s `Discovery.ServerVersion()`），更新 `ok_clusters.Status`。
6. 【v2】删除 / 更新集群时，写锁下从池中摘除旧客户端并重建，同时触发 Casbin 级联清理（见 5.4）。

**请求路由**：每次前端 HTTP 请求在 Header 携带 `X-Cluster-ID`；WebSocket 因无法自定义 Header，改用 query 参数 `?cluster_id=`（见第 8 章）。

## 4. 权限与隔离模型设计（核心卖点）【v2 重写】

v1 的问题：4.1 声称“预设角色”，4.2 却给出扁平 `p` 策略，二者无法对应。v2 明确采用 **RBAC with Domains + 角色分组(g) + 资源分组** 的完整模型。

### 4.1 Casbin model.conf

```ini
[request_definition]
r = sub, dom, obj, act

[policy_definition]
p = sub, dom, obj, act

[role_definition]
g = _, _, _          # 角色绑定支持 domain: g(user, role, domain)

[policy_effect]
e = some(where (p.eft == allow))

[matchers]
m = g(r.sub, p.sub, r.dom) \
    && keyMatch(r.dom, p.dom) \
    && resMatch(r.obj, p.obj) \
    && (r.act == p.act || p.act == "*")
```

* `g(user, role, dom)`：用户在某 domain 下绑定某角色，角色权限不会泄漏到其他 domain。
* `keyMatch`：支持 `cluster_g:*` 通配该集群所有 NS。
* 【v2】`resMatch`（自定义函数）：把具体资源映射到资源组，使策略可写 `workloads` 而请求传 `pods`（见 4.3）。

### 4.2 四种预设角色 → 策略（`p` + `g`）

角色权限以 `p` 规则定义，用户通过 `g` 规则绑定到角色。

| 角色 | p 策略 (sub, dom, obj, act) 示例 |
| --- | --- |
| `Cluster-Admin` | `p, Cluster-Admin, *, *, *` |
| `Cluster-Viewer` | `p, Cluster-Viewer, *, *, read` |
| `NS-Editor` | `p, NS-Editor, *, workloads, write` / `... network, write` / `... config, reveal` / `... pods, exec` |
| `NS-Viewer` | `p, NS-Viewer, *, *, read` |

绑定示例（`g`）：

| g 规则 | 含义 |
| --- | --- |
| `g, user_a, Cluster-Admin, cluster_f` | user_a 是集群 F 的集群管理员 |
| `g, user_a, NS-Editor, cluster_g:dev-ns` | user_a 在集群 G 的 dev-ns 可读写 |
| `g, user_b, NS-Viewer, cluster_g:test-ns` | user_b 在集群 G 的 test-ns 只读 |

> 【v2】关键差异：`Cluster-Admin/Viewer` 的 domain 是 `cluster_f`（keyMatch 时等价 `cluster_f` 自身的集群级资源）；NS 角色的 domain 是 `cluster_g:dev-ns`。集群级资源（Node/PV/CRD）的请求 domain 为 `cluster_f`，绝不会被 NS 角色命中。

### 4.3 资源分组与动作映射【v2 新增】

**资源组**（`resMatch` 维护的映射，可配置化）：

| 资源组 | 具体 K8S 资源 |
| --- | --- |
| `workloads` | deployments, statefulsets, daemonsets, pods, jobs, replicasets |
| `network` | services, ingresses, endpoints |
| `config` | configmaps, secrets, persistentvolumeclaims |
| `cluster` | nodes, persistentvolumes, namespaces, customresourcedefinitions |

**动作（act）**：v1 的二元 read/write 不足以描述 exec/reveal，v2 扩展为：

| act | HTTP / 触发方式 | 说明 |
| --- | --- | --- |
| `read` | GET | get / list / watch |
| `write` | POST / PUT / PATCH / DELETE | 创建/更新/删除（PATCH 即 apply）|
| `exec` | WebSocket(exec) | 【v2】WebSSH 进容器，独立动作，**不被 write 覆盖** |
| `reveal` | POST `/secrets/:name/reveal` | 【v2】Secret 明文揭示，独立动作 + 审计 |

## 5. 功能模块详细设计

### 5.1 平台全局初始化与集群管理

* **Admin 自动就绪**：首次启动检测 `ok_users` 为空时，自动创建 `admin`（随机初始密码打印到启动日志，`MustReset=true`，首登强制改密）。
* **多集群绑定**：表单录入新集群，入池前用 `Discovery` 验证连通性与最小权限，成功才落库 + 入池。

### 5.2 全局权限指派中心（Admin 专属）

* **用户管理**：Admin 创建/禁用普通用户。
* **授权矩阵配置**（级联选择器）：
  * 层级 1：选目标集群。
  * 层级 2：选范围（A 整个集群 / B 指定命名空间，多选）。
  * 层级 3：选角色（`Cluster-Admin` / `NS-Editor` ...）。
  * 提交后后端写入对应 `g` 绑定规则（而非直接写 `p`）。

### 5.3 K8S 全资源运维看板（多租户动态流）

资源请求由顶栏 **【集群下拉框】** + **【Namespace 下拉框】** 联动驱动。

* **NS 下拉数据权限**：`GET /api/v1/clusters/:id/namespaces` 【v2】由中间件注入的「可见 NS 列表」过滤，只返回用户在该集群有任意角色绑定的 NS（Cluster 级角色返回全部）。
* **工作负载**：Deployments / StatefulSets / DaemonSets / Pods / Jobs。
  * **WebSSH**：仅当用户在该 NS 拥有 `exec` 动作（即 NS-Editor 及以上）时 UI 才显示“终端”按钮；后端二次鉴权（见第 8 章）。
  * **实时日志**：`kubectl logs -f` 经 WebSocket 推流到 xterm.js，鉴权动作为 `read`。
* **网络**：Services / Ingresses，可视化展示 Ingress 域名 → Service → 后端 Pod 存活。
* **存储与配置**：ConfigMaps / Secrets / PVC。
  * 【v2】Secret 列表默认 `******`。揭示明文走独立接口 `POST /secrets/:name/reveal`，要求 `reveal` 动作（NS-Editor 及以上），**每次揭示写审计日志**。（注：K8S Secret 本质是 base64，此处是“按 RBAC 门控的明文展示 + 留痕”，非加解密。）

### 5.4 级联清理【v2 新增】

* **删除集群**：DB 删除 + 连接池摘除 + 删除所有 `v1 LIKE 'cluster_x%'` 的 Casbin 规则（事务保证一致）。
* **删除用户**：DB 删除 + 删除所有 `v0 = user` 的 `g` 规则。

## 7. 安全核心：后端中间件链【v2 重写】

请求依次经过 **JWT 认证中间件 → RBAC 鉴权中间件**。鉴权中间件核心伪代码：

```go
func RBACAuthMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        userID := c.MustGet("user_id").(string) // 由 JWT 中间件注入
        clusterID := c.GetHeader("X-Cluster-ID")
        if clusterID == "" || !ClusterExists(clusterID) {
            abort(c, 400, "无效的集群标识"); return
        }
        if IsSystemAdmin(userID) { c.Next(); return }

        // 【v2 关键修复 #1】namespace 单一可信来源:
        // 统一从 path 解析(写操作), 列表类从 query; 严禁 body 与鉴权值分离。
        // 解析后将 namespace 注入 context, handler 下发 K8S 时必须用同一个值,
        // 并在下发前强制覆盖 body.metadata.namespace, 杜绝参数混淆越权。
        resource := parseK8sResource(c) // -> "pods"/"nodes"/"secrets"...
        namespace := resolveAuthNamespace(c, resource) // 唯一裁决点
        c.Set("auth_namespace", namespace)

        action := parseAction(c) // GET->read, POST/PUT/PATCH/DELETE->write
        // exec / reveal 由各自专用 handler 显式声明, 不走通用映射

        // 构建 domain
        domain := clusterID
        if namespace != "" {
            domain = fmt.Sprintf("%s:%s", clusterID, namespace)
        }

        allowed, _ := enforcer.Enforce(userID, domain, resource, action)
        if !allowed {
            // 【v2 关键修复 #2】受控的集群级只读放行:
            // 仅当 (集群级资源 && read && 资源属于"可跨NS聚合的白名单") 才放行,
            // 且必须配合 handler 按"可见NS列表"过滤结果, 否则拒绝。
            if isClusterScope(namespace) && action == "read" &&
               isAggregatableRead(resource) {
                visibleNS := ListVisibleNamespaces(userID, clusterID)
                if len(visibleNS) > 0 {
                    c.Set("visible_ns", visibleNS) // handler 必须据此过滤
                    audit(c, userID, clusterID, resource, action, "allow")
                    c.Next(); return
                }
            }
            audit(c, userID, clusterID, resource, action, "deny")
            abort(c, 403, "鉴权失败：无该集群或命名空间的操作权限"); return
        }
        if action == "write" { audit(c, userID, clusterID, resource, action, "allow") }
        c.Next()
    }
}
```

**三处修复要点**：

1. **#1 namespace 单一可信来源**：鉴权用的 namespace 与下发 K8S 的 namespace 必须同源；下发前强制覆盖 body 里的 namespace 字段，封堵 `?namespace=A` + `body: namespace=B` 的参数混淆越权。
2. **#2 受控集群级只读**：v1 无条件放行的“降级检查”收紧为「白名单资源 + 强制结果过滤」，handler 拿 `visible_ns` 裁剪返回集，NS-Viewer 不会看到未授权空间的数据。
3. **#3 WebSSH/WS 独立鉴权**：见第 8 章，`exec` 为独立动作，不被 `write` 覆盖。

## 8. WebSocket 鉴权（WebSSH / 日志流）【v2 新增】

浏览器原生 WebSocket 无法自定义 Header，v1 依赖 `X-Cluster-ID` Header 的方案在此失效。v2 方案：

* **连接参数**：`wss://.../api/v1/exec?cluster_id=cluster_g&namespace=dev-ns&pod=nginx&container=app&token=<jwt>`
  （token 经 query 传递，连接建立后立即校验并从日志/审计中脱敏）。
* **升级前鉴权**：在 `Upgrade` 之前完成
  1. JWT 校验 → `user_id`；
  2. 解析 `cluster_id` / `namespace`；
  3. WebSSH 校验 `exec` 动作、日志流校验 `read` 动作：`enforcer.Enforce(userID, "cluster_g:dev-ns", "pods", "exec")`；
  4. 不通过直接拒绝握手（403），不进入 stream。
* **执行**：通过 `ClusterClient.Config` + `remotecommand.NewSPDYExecutor` 建立流；WebSSH 会话**强制写审计日志**（user / cluster / ns / pod / container / 起止时间）。

## 9. v1 → v2 变更对照

| # | v1 问题 | v2 修复 |
| --- | --- | --- |
| 1 | 鉴权 namespace 取自 query，可与 body 不一致 → 越权 | namespace 单一可信来源 + 下发前强制覆盖 (第 7 章) |
| 2 | “降级检查”无条件放行集群级 read → 跨 NS 泄露 | 白名单 + `visible_ns` 强制过滤 (第 7 章) |
| 3 | WebSSH 依赖 Header，WS 无法携带 → 跑不通 | query token + 升级前鉴权 (第 8 章) |
| 4 | WebSSH 复用 write 权限，过粗 | 独立 `exec` 动作 (4.3 / 第 8 章) |
| 5 | 预设角色与扁平 p 策略对不上 | model.conf + `g` 角色绑定 + 资源组 (第 4 章) |
| 6 | 单一 Clientset 撑不起 CRD/全资源 | typed+dynamic+discovery+RESTMapper (2.2) |
| 7 | 连接池非线程安全、重启不重建 | RWMutex + 启动重建 + 定时探活 (2.2) |
| 8 | resource/action 映射缺失、漏 PATCH | 资源组映射 + 动作扩展 (4.3) |
| 9 | 无审计、无认证、加密无密钥管理 | 审计表 + JWT + AES-GCM/KMS (3.4 / 1.2) |
| 10 | 删除集群/用户残留策略与连接 | 级联清理事务 (5.4) |
```
