# OmniKube 多集群多租户 K8S 管控平台 - 功能设计书

## 1. 项目概述

### 1.1 项目简介

OmniKube 是一个基于 Golang 和 React 构建的企业级**多集群、多租户 Kubernetes 可视化管理平台**。平台核心解决“用户全局统一管理”、“多 K8S 集群动态托管绑定”以及“精细到 Namespace（命名空间）级别的资源可见性与操作权限隔离”三大痛点。用户只需在 OmniKube 注册一次，即可由系统管理员（Admin）按需分配不同集群、不同 Namespace 的管理或只读权限。

### 1.2 技术栈选型

* **后端 (Backend):** Golang + Gin (高效路由) + Client-go (K8S 原生交互) + Casbin (基于 Domain 的 RBAC 权限引擎) + GORM (数据库 ORM)
* **前端 (Frontend):** React + Ant Design (组件库) + Axios (请求封装) + xterm.js (WebSSH 终端)
* **数据存储:** MySQL / PostgreSQL (保存用户信息、集群 Kubeconfig 及权限策略)

## 2. 核心架构与多集群动态连接机制

### 2.1 逻辑架构

OmniKube 独立于 K8S 集群运行。它作为统一网关，后端内存中维护一个**集群连接池 (Cluster Connection Pool)**。

```
                   ┌─────────────────────────────────────────┐
                   │            OmniKube 前端 (React)        │
                   └────────────────────┬────────────────────┘
                                        │ (HTTP / WebSocket)
                                        ▼
                   ┌─────────────────────────────────────────┐
                   │            OmniKube 后端 (Go)           │
                   │  ┌───────────────────┐  ┌────────────┐   │
                   │  │ ClusterRBAC 中间件 │  │ Casbin 引擎 │   │
                   │  └─────────┬─────────┘  └──────┬─────┘   │
                   │            │                   │         │
                   │            ▼                   │         │
                   │  ┌───────────────────┐         │         │
                   │  │ 集群动态连接池     │◄────────┘         │
                   │  │ (Client-go Map)   │                   │
                   │  └─────────┬─────────┘                   │
                   └────────────┼─────────────────────────────┘
                                │ (动态路由切换)
         ┌──────────────────────┼──────────────────────┐
         ▼                      ▼                      ▼
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│  K8S 集群 F     │    │  K8S 集群 G     │    │  K8S 集群 H     │
└─────────────────┘    └─────────────────┘    └─────────────────┘

```

### 2.2 动态连接池实现原理

1. Admin 在 Web 界面添加新集群，输入 `ClusterName` 并粘贴 `Kubeconfig` 文本。
2. 后端通过 `client-go` 的 `tools/clientcmd` 解析该文本，生成 `*kubernetes.Clientset`。
3. 将 `Clientset` 存入内存中的 Map：`var ClusterPool = make(map[string]*kubernetes.Clientset)`。
4. 每次前端请求（如获取 Pod 列表）时，必须在 Header 中携带 `X-Cluster-ID`。后端中间件根据该 ID 从 Map 中获取对应的 `Clientset` 与目标集群交互。

## 3. 数据库模型设计 (Database Schema)

系统核心包含 4 张表，用于支撑多集群下的多租户全局权限体系。

### 3.1 用户表 (`ok_users`)

存储平台全局唯一的用户信息。

```go
type User struct {
    ID       uint   `gorm:"primaryKey"`
    Username string `gorm:"unique;not null;size:50"`
    Password string `gorm:"not null;size:100"` // 密码 Hash
    IsAdmin  bool   `gorm:"default:false"`     // 是否为系统级超级管理员
}

```

### 3.2 集群表 (`ok_clusters`)

存储托管的 K8S 集群凭证。

```go
type Cluster struct {
    ID         string `gorm:"primaryKey;size:50"` // 英文唯一标识，如 cluster_hk
    Name       string `gorm:"not null;size:100"`  // 别名，如 香港生产集群
    Kubeconfig string `gorm:"type:text;not null"` // 加密存储的 Kubeconfig 文本
    Status     string `gorm:"size:20;default:'Healthy'"` // Healthy / Unreachable
    CreatedAt  time.Time
}

```

### 3.3 权限策略表 (`casbin_rule`)

使用 Casbin 默认的表结构，通过 `v1 (Domain)` 字段来实现“集群:命名空间”的三维隔离。

* **字段对应关系:** `v0` (用户名/角色), `v1` (Domain范围), `v2` (资源), `v3` (动作)。

## 4. 权限与隔离模型设计（核心卖点）

权限严格划分为**集群级（非名称空间资源）**与**命名空间（NS）级（名称空间资源）**，并拆解为**数据可见权限**与**操作权限**。

### 4.1 权限粒度定义

平台预设四种标准角色，通过 Casbin 策略控制：

1. **`Cluster-Admin` (集群管理员):**
* **可见性:** 可见该集群下的所有资源和所有 Namespace。
* **操作:** 可操作集群级资源（Nodes, PV, CRD）及所有 NS 的写/删动作。


2. **`Cluster-Viewer` (集群只读员):**
* **可见性:** 可见该集群的所有 Node 状态、PV 及所有 NS 资源。
* **操作:** 全局只读，无法执行创建、删除或修改。


3. **`NS-Editor` (空间读写员):**
* **可见性:** 仅可见被授权的 `ClusterID:Namespace` 空间。
* **操作:** 可在当前 NS 内创建、修改、删除 Workloads (Pod, Deployment) 和 Network (Service, Ingress)。


4. **`NS-Viewer` (空间只读员):**
* **可见性:** 仅可见被授权的 `ClusterID:Namespace` 空间。
* **操作:** 仅能执行 `get`, `list`, `watch`，无法修改任何资源，无法进入 WebSSH。



### 4.2 Casbin 策略矩阵示例 (DB 存储形态)

| p_type | v0 (Subject) | v1 (Domain 范围) | v2 (Resource) | v3 (Action) | 说明 |
| --- | --- | --- | --- | --- | --- |
| `p` | `user_a` | `cluster_f` | `*` | `*` | User A 是集群 F 的全局管理员 |
| `p` | `user_a` | `cluster_g:dev-ns` | `workloads` | `write` | User A 在集群 G 的 dev-ns 空间有读写权限 |
| `p` | `user_b` | `cluster_g:test-ns` | `workloads` | `read` | User B 在集群 G 的 test-ns 空间仅有只读权限 |

## 5. 功能模块详细设计

### 5.1 平台全局初始化与集群管理

* **Admin 自动就绪:**
* 系统首次部署启动时，若检测到 `ok_users` 表为空，自动在数据库中创建账号 `admin`，密码预设（首次登录强制修改）。
* Admin 登录后直接进入“集群管理”面板。


* **多集群绑定:**
* 提供表单，支持 Admin 动态录入新集群、测试连接性（调用 `client-go` 的 `DiscoveryClient` 验证权限）。



### 5.2 全局权限指派中心（Admin 专属）

* **用户管理:** Admin 可一键创建普通用户账号（User A、User B）。
* **授权矩阵配置:**
* 选择一个全局用户 -> 点击“指派权限” -> 弹出级联选择器。
* **层级 1:** 选择目标集群。
* **层级 2:** 选择权限范围（选项 A：整个集群；选项 B：指定命名空间，支持多选）。
* **层级 3:** 赋予角色（`Cluster-Admin` / `NS-Editor` 等）。
* 提交后，后端自动向 Casbin 策略表插入相应的 `p` 规则。



### 5.3 K8S 全资源运维看板（多租户动态流）

所有资源请求由前端顶栏的 **【集群下拉框】** 和 **【Namespace 下拉框】** 联动驱动。

#### 1. 工作负载 (Workloads)

* **管理资源:** Deployments, StatefulSets, DaemonSets, Pods, Jobs。
* **数据权限拦截:** 普通用户切换集群时，NS 下拉框调取的接口 `/api/v1/clusters/:id/namespaces` 只会返回其在 Casbin 中拥有 `read` 策略的 NS。
* **操作权限控制:**
* **WebSSH 容器终端:** 只有在当前 NS 拥有 `write` 权限的角色，UI 才会显示 “终端” 按钮。点击后通过 WebSocket 连通后端，利用 `client-go/tools/remotecommand` 实现流式命令交互。
* **实时日志:** 封装 `kubectl logs -f`，通过 WebSocket 流式推送到前端 `xterm.js` 模拟流。



#### 2. 网络与路由 (Network)

* **管理资源:** Services, Ingresses。
* **可视化展示:** 页面直观展示 Ingress 配置的域名、转发的 Service 以及后端 Pod 的存活状态。

#### 3. 存储与配置 (Storage & Config)

* **管理资源:** ConfigMaps, Secrets, PVC。
* **安全脱敏:** Secret 列表中的明文数据（如数据库密码）默认处于 `******` 遮蔽状态，仅 `NS-Editor` 以上级别点击方可解密看明文。

## 6. 前端页面解构 (React 路由与交互)

前端采用标准的侧边栏+顶栏联动布局。顶栏持有全局状态：`currentCluster` 和 `currentNamespace`。

```
/login                          # 登录页 (全局单点登录)
/clusters                       # 超级管理员专用：集群添加与连接状态看板
/users                          # 超级管理员专用：用户管理与权限矩阵指派

# 以下页面全部依赖顶栏的 Cluster 与 NS 状态
/pages/dashboard                # 资源总览 (展示当前集群/当前 NS 的 CPU、内存配额及拓扑)
/pages/workloads/deployments    # Deployment 列表与 YAML 编辑
/pages/workloads/pods           # Pod 列表、日志查看、WebSSH 抽屉弹出
/pages/networking/services      # Service 管理
/pages/storage/configmaps       # 配置项管理

```

## 7. 安全核心：后端中间件拦截逻辑

所有发往 K8S 的请求，由后端 `RBACAuthMiddleware` 统一拦截。其核心伪代码如下：

```go
func RBACAuthMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        // 1. 获取全局唯一用户
        userID := c.MustGet("user_id").(string)
        if IsSystemAdmin(userID) { c.Next(); return } // 系统级超级管理员直接放行
        
        // 2. 获取当前操作的目标集群和目标空间
        clusterID := c.GetHeader("X-Cluster-ID")
        namespace := c.Query("namespace") // 若为集群级资源，此项为空
        
        // 3. 解析当前 HTTP 请求对应的资源和动作
        resource := parseK8sResource(c.Path) // 例如: "pods", "deployments", "nodes"
        action := parseHttpAction(c.Method)  // GET-> "read", POST/PUT/DELETE-> "write"
        
        // 4. 构建 Casbin Domain 域
        var domain string
        if namespace == "" {
            domain = clusterID // 集群级权限校验
        } else {
            domain = fmt.Sprintf("%s:%s", clusterID, namespace) // NS 级权限校验
        }
        
        // 5. 权限判定
        allowed, _ := casbinEnforcer.Enforce(userID, domain, resource, action)
        if !allowed {
            // 降级检查：如果是集群级资源，但用户在该集群下的某些 NS 有权限，允许其执行某些特定的“只读”感知
            if namespace == "" && action == "read" && HasSubNamespacePermission(userID, clusterID) {
                c.Next()
                return
            }
            c.JSON(403, gin.H{"error": "鉴权失败：您没有该集群或命名空间的操作权限"})
            c.Abort()
            return
        }
        c.Next()
    }
}

```
