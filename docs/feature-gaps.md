# OmniKube 功能查漏补缺 / Roadmap

> 生成日期:2026-07-01
> 目的:盘点当前已实现能力,列出企业级多集群 K8s 管理平台还可以补齐的功能,按优先级给出建议。

---

## 一、当前已实现能力(基线)

| 模块 | 已有能力 |
|------|----------|
| **认证** | 账号密码登录、JWT、修改密码、管理员重置用户密码 |
| **RBAC 权限** | Casbin + domain 模型;全局权限 + 「每集群 × 每资源 × 操作」矩阵;预置系统角色(已 i18n);角色增删改查、复制角色 |
| **用户管理** | 增删改查、启用/禁用、分配角色 |

## 二、缺失能力清单(按优先级)

### 🔴 P0 —— 企业平台的明显短板,建议优先补

#### 1. 审计日志中心(最大短板)
- **现状**:审计只覆盖「Secret 揭示」;创建/更新/删除资源、登录、改权限、集群变更等**都没有落审计**;且**没有查询接口和界面**。
- **补齐**:
  - 中间件级审计:对所有写操作(create/update/delete/发布/权限变更/登录登出)自动记录「谁 / 何时 / 在哪个集群+命名空间 / 对什么资源 / 做了什么 / 成功失败」。
  - 审计查询页:按用户、动作、资源、时间范围、集群过滤;支持导出 CSV。
  - 合规价值高,企业采购几乎必查。

#### 2. 监控指标(资源水位)
- **现状**:仪表盘只有「数量统计」,没有 CPU/内存实时用量。
- **补齐**:
  - 接入 metrics-server:Pod/Node 的 CPU、内存 top。
  - 节点资源水位条、命名空间用量、工作负载资源用量趋势(可选接 Prometheus)。
  - Pod 列表增加 CPU/Mem 列;Node 页展示 allocatable / requested / 水位。

#### 3. 工作负载运维操作(日常高频)
- **现状**:只能改镜像和删除,缺少常见运维动作。
- **补齐**:
  - **伸缩副本**(Scale):列表/详情一键调副本数。
  - **重启**(rollout restart):滚动重启 Deployment/StatefulSet/DaemonSet。
  - **回滚**(rollback):基于历史 revision 回滚,发布记录页直接「回滚到此版本」。
  - **暂停/恢复**发布(pause/resume rollout)。
  - **Pod 操作**:删除/驱逐单个 Pod、查看 Pod 所属节点、日志下载。
  - **Describe 视图**:kubectl describe 风格的聚合详情(事件+状态+条件)。

### 🟠 P1 —— 提升平台完整度

#### 4. 更多资源类型
- 原生 RBAC:ServiceAccount / Role / RoleBinding / ClusterRole(与平台自身 RBAC 区分开)。
- 弹性:HorizontalPodAutoscaler(HPA)。
- 稳定性:PodDisruptionBudget。
- 网络:NetworkPolicy、Endpoints。
- 配额:ResourceQuota / LimitRange(配合多租户)。
- ReplicaSet(只读,排障用)。
- StorageClass(目前被引用但缺独立管理页)。
- **CRD / 自定义资源**:通用列表+YAML 编辑,支持 Operator 生态。

#### 5. 事件中心 & 资源健康
- 每个资源详情页的「Events」标签(当前只有仪表盘全局事件)。
- 集群巡检:节点 NotReady、Pod CrashLoopBackOff、镜像拉取失败等异常聚合。

#### 6. 通知与告警增强
- **测试推送**按钮:配好 webhook 立即发一条测试消息验证(之前已向用户提议)。
- 通知**事件类型可选**:不只发布,支持「Pod 异常 / 节点宕机 / 资源超配」等告警推送。
- **邮件通知**渠道;自定义消息模板;@ 指定人。
- 集群健康掉线主动告警(而非仅发布时通知)。

#### 7. 多租户 / 项目分组
- 把命名空间归组为「项目/团队」,按项目授权与查看。
- 项目级资源配额与用量看板。

### 🟡 P2 —— 增值 & 体验

#### 8. 登录与账号安全
- SSO:OIDC / LDAP / OAuth2 单点登录对接。
- 双因素认证(2FA / TOTP)。
- 密码策略(复杂度、有效期)、登录失败锁定、在线会话管理与强制下线。
- **API Token / 机器人账号**:给 CI/CD 或自动化签发受限 Token(区别于人用账号)。

#### 9. 发布能力增强
- 发布**审批流**(申请 → 审批 → 执行),关键环境卡点。
- 发布前 **YAML diff 预览**(改动可视化)。
- 灰度 / 金丝雀发布(可选,依赖 Ingress/Service 权重)。
- 镜像仓库集成:填镜像时**拉取 tag 列表**下拉选择(Harbor/Docker Registry)。

#### 10. Helm / 应用编排
- Helm Release 管理:安装、升级、回滚、values 编辑、chart 仓库。
- 或应用模板(把多资源打包成一次发布)。

#### 11. 效率与体验
- **全局资源搜索**:跨命名空间/资源类型快速定位。
- **批量操作**:批量删除、批量重启、批量打标签。
- YAML 从文件导入 / 导出下载。
- Pod **文件浏览器**:容器内文件上传下载。
- **暗黑模式**(Dark Mode)。
- 收藏 / 常用入口、最近访问。
- exec 终端 / 日志流断线重连。

## 三、建议落地顺序

1. **审计日志中心**(P0-1)—— 合规刚需,基础设施已就位,ROI 最高。
2. **工作负载运维操作**(P0-3)—— 伸缩/重启/回滚,日常高频,改动集中在 resource handler。
3. **监控指标接入 metrics-server**(P0-2)—— 仪表盘和列表体验质变。
4. **通知测试按钮 + 告警事件类型**(P1-6)—— 承接刚做完的 webhook 功能,小步快跑。
5. 之后按需推进更多资源类型(P1-4)、多租户(P1-7)、SSO/API Token(P2-8)。

---

## 四、备注

- 现有代码结构清晰(handler / rbac / notify / audit 分层),上述多数功能可在现有分层内扩展,无需重构。
- 「资源运维操作」和「更多资源类型」主要落在 `backend/internal/handler/resource.go` + 前端 `components/editor` 与各资源页,增量成本低。
- 「审计中心」需新增查询 handler + 前端页面,并在写操作路径统一埋点(建议做成中间件)。
